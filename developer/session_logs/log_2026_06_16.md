## Validación y Decisiones del Usuario (16 de Junio de 2026)

- **Tecnología Frontend**: **React + TypeScript + Vite**. Se descarta Vue por decisión del usuario (React le resulta más familiar).
- **Desktop**: **Tauri**. Se confirma su uso para empaquetar la app de escritorio y ejecutar procesos locales de renderizado.
- **Base de Datos**: **SQLite**. Se confirma su uso local en el servidor y escritorio por su simplicidad y nula necesidad de configuración.
- **Infraestructura del Servidor**: Se opta por desarrollar la API de backend completamente en **Node.js**. Durante el desarrollo y pruebas iniciales, el servidor se ejecutará en **local**. El código se organizará en el repositorio de GitHub para permitir despliegues sencillos en VPS o servicios en la nube en el futuro por parte del usuario o la comunidad. No se utilizará PHP.
- **Retrocompatibilidad**: No se requiere compatibilidad hacia atrás con la versión 1 (v1). Sin embargo, a partir de la v2.0.0, todo el desarrollo debe garantizar compatibilidad hacia adelante y hacia atrás de los archivos del proyecto y esquemas.
- **Portabilidad de Archivos (.cdc2)**: Para resolver la portabilidad de imágenes entre ordenadores de diferentes usuarios, el formato de archivo de guardado `.cdc2` será un **contenedor ZIP** renombrado. Este archivo empaquetará tanto el archivo JSON de configuración del proyecto (`project.json`) como una subcarpeta `assets/` con todas las imágenes locales importadas, referenciadas bajo un protocolo virtual personalizado `asset://`.
- **Primera Especificación**: Módulo de maquetación e importación en bloque de imágenes de cartas sobre lienzos de tamaño configurable.

- **Resultado de la Fase**: La Fase 1 está **completada y validada**. Compila con éxito y supera los 5 tests automáticos en Vitest.
- **Importación de Traseras en Bloque**: Añadido el manejador `handleTraseraImportBloque` y un botón dedicado en la interfaz. Permite cargar un lote de archivos de reverso y asociarlos de forma secuencial ordenada uno a uno en la lista de cartas.
- **Flujo de Reversos Unificado**: Se reemplazaron los modos rígidos de reverso por un interruptor "Generar Reversos (Doble Cara)". Si está activo, el usuario puede subir una trasera común por defecto para todo el mazo, y a la vez subir traseras individuales específicas por carta (las cuales sobrescriben la común). Si se elimina la trasera individual, la carta vuelve a heredar la común automáticamente.
